# 面向小型操作系统的分析比对智能体系统 —— 设计方案

> 项目代号：**osAgent**
> 文档版本：v1.0 （初稿）
> 适用对象：全国大学生操作系统比赛（OS Kernel 赛道）评审与组委会
> 数据底座：[`collected-data.xlsx`](collected-data.xlsx)（2021–2025 共 **168** 个优秀作品仓库）

---

## 1. 背景与问题陈述

| 维度 | 现状 | 痛点 |
|---|---|---|
| 数据规模 | 2021(21) / 2022(30) / 2023(27) / 2024(38) / 2025(52)，五届累计 168 个 OS Kernel 仓库，2026 年预计再新增 60+ | 仓库数量指数级增长，人工评审难以覆盖 |
| 代码形态 | Rust / C 为主，RISC-V 架构，多为 RCore / xv6-k210 / UCore 的二次开发或独立微内核 | 同源代码差异巨大，"看上去像"未必是抄袭，"看上去不像"也未必是创新 |
| 评审需求 | 1) 快速理解一个新仓库的设计；2) 与历史 168+ 作品做查重 / 创新点比对；3) 沉淀技术演进路径 | 现有 MOSS / Simian 等文本查重工具对内核语义无感知，幻觉、漏报、误报频发 |

**核心命题**：构建一套以 **LLM Agent + RAG + 静态分析** 为核心的「小型操作系统智能分析比对系统」，**输入一个仓库 URL，输出两份文档**：

1. **结构化描述文档**（Project Card）——对人类友好、抗幻觉
2. **创新点 & 查重对比文档**（Diff Report）——与历史库精准比对

## 2. 设计目标与非目标

### 2.1 目标（Must Have）
- **G1 抗幻觉**：所有关键结论必须可追溯到具体源码行号（line-level citation）
- **G2 精准比对**：相似度判定不仅给分数，还给"哪个文件、哪个函数、哪段提交历史"的证据链
- **G3 可复现**：同一仓库多次分析，结论稳定（temperature=0 + 缓存）
- **G4 可扩展**：新增一届比赛、新增一个仓库，增量入库 < 30 min
- **G5 国产化**：LLM 首选 **DeepSeek-V3 / Qwen3 / GLM-4.6** 等国产开源模型

### 2.2 非目标（Out of Scope，v1 不做）
- 不做编译运行级别的动态测试（留给比赛官方测试床）
- 不做完整的 IDE 插件，只提供 Web Dashboard + CLI
- 不替代人工评审，只做"评审副驾驶"

### 3.1 模块职责一览

| 模块 | 类型 | 职责 | 关键技术 |
|---|---|---|---|
| **Repo Ingest Agent** | Agent | 克隆仓库、识别 build 系统、抽取 README/commit log | `git`, `cloc`, GitPython |
| **Static Analyzer (MCP Server)** | MCP | 提供"函数级语义"工具：调用图、内核子系统识别、syscall 表抽取 | `tree-sitter`, `clangd`, `rust-analyzer`, `Joern` |
| **Semantic Indexer** | 离线 | 把 168+ 仓库切块、向量化、入库 | `BGE-M3`, `Milvus`, AST-aware chunking |

| **Report Writer Agent** | Agent | 生成 Project Card / Diff Report，每条结论附引用 | Skills（模板化写作技能） |
| **Orchestrator** | 主 Agent | ReAct 调度，处理人类反馈循环 | LangGraph / 自研 |

## 4. 关键技术选型

### 4.1 LLM 选型（国产 / 开源优先）

| 角色 | 首选 | 备选 | 理由 |
|---|---|---|---|
| 主推理 / 调度 | **DeepSeek-V3.2** | Qwen3-235B | 长上下文、工具调用稳定 |
| 代码理解 | **Qwen3-Coder-30B-A3B** | DeepSeek-Coder-V2 | 对 Rust / C 内核代码识别准确 |
| 嵌入 | **BGE-M3** | Qwen3-Embedding-8B | 中英 + 代码混合，1024 维 |
| 重排 | **BGE-Reranker-v2-M3** | — | 提升 RAG 精度 |

> 部署：通过 **vLLM** + **SGLang** 自托管；对评审场景敏感数据全程不出内网。

### 4.2 Agent 框架
- **LangGraph**（推荐）：状态机式 Agent，便于复现与调试
- 工具协议统一用 **MCP**（Model Context Protocol），所有静态分析能力都以 MCP Server 暴露，便于 Claude Code / Cursor / 自研客户端复用

### 4.3 抗幻觉的核心机制 ⭐
1. **强制引用（Citation Required）**：Report Writer 输出的每一段结论必须带 `[file.c:L120-L145]` 引用，未引用直接拒采
2. **代码先于文字**：先用 Static Analyzer 抽出"事实表"（调用图、syscall 列表、模块文件清单），LLM 只允许在事实表上"复述+解释"
3. **双模型交叉校验**：关键结论（如"实现了 SMP"）由 DeepSeek 和 Qwen 双模型独立判定，不一致则降级到"需人工确认"
4. **温度 0 + 结构化输出（JSON Schema）**

### 5.2 Diff Report（创新点 & 查重对比文档）

```markdown
# UltraOS-2026 vs 历史库 Top-3 对比

## 综合判定
- 整体相似度：**42%**（中等）
- 结构同源最近邻：**UltraOS-2021**（同校系传承，置信度 95%）
- 创新点：**3** 项；**疑似抄袭**：0 项；**值得复核**：1 项

## 1. 与 UltraOS-2021 的对比（同源演进，非抄袭）
| 子系统 | 2021 | 2026 | 判定 |
|---|---|---|---|
| 调度器 | 优先级队列 | CFS（红黑树） | ✅ 重大演进，引用 sched/cfs.rs:1-300 |
| 文件系统 | 自研 FAT32 | ext2 + VFS | ✅ 架构重构 |
| 内存 | buddy | buddy + slab | ⚠️ 增量改进 |
> 同源识别证据：commit 历史中含 `Initial commit from UltraOS-2021`（commit a1b2c3d）

## 2. 与 XX-2024 的对比（疑似函数级雷同）⚠️ 需复核
- 文件：`mm/slab.rs` 与 XX-2024 的 `kernel/mm/slab.rs` MinHash 相似度 **0.91**
- LLM 语义判定：**两段代码逻辑同构，但变量重命名、注释翻译**
- 证据并排展示（见附录 A）

## 3. 真正的创新点（去重后）
1. 引入异步 IO 框架（首个在赛道中使用 io_uring 风格的实现）
2. ...
```

### 5.3 处理流水线（伪代码）

```python
def analyze_repo(url: str) -> Tuple[ProjectCard, DiffReport]:
    # 1. Ingest
    repo = ingest_agent.clone_and_scan(url)
    
    # 2. 事实抽取（先于 LLM）
    facts = static_analyzer.extract(repo)   # MCP 工具调用
    #   facts: 调用图、syscall 表、模块清单、commit 时间线、loc 统计
    
    # 3. 生成 Project Card（基于 facts，禁止脱离事实表）
    card = writer_agent.build_card(facts, repo.readme)
    
    # 4. 多路召回
    candidates = retrieval_agent.search(
        embed=facts.kernel_design_embedding,
        bm25=repo.readme + facts.module_names,
        call_graph_minhash=facts.cg_minhash,
        top_k=5,
    )
    
    # 5. 逐一深度比对
    diffs = [compare_agent.deep_compare(repo, c) for c in candidates]
    
    # 6. 生成 Diff Report（双模型交叉校验）
    report = writer_agent.build_diff_report(diffs, models=["deepseek", "qwen"])
    
    return card, report
```

## 6. 历史库索引方案（离线一次性 + 增量）

### 6.1 数据治理（基于 [`collected-data.xlsx`](collected-data.xlsx)）
- 168 个仓库批量克隆，失效链接打 `unreachable` 标签
- 每个仓库存储三层物化：
  1. **原始代码快照**（按 commit hash 锁定）
  2. **事实表**（JSON，约 50KB/仓库）
  3. **向量切片**（约 500–2000 块/仓库）

### 6.2 切块策略（AST-aware Chunking）
- 不按字符切，按 **函数 / 模块** 切（tree-sitter 解析）
- 每块附带元数据：`{repo, year, file, func, callers, callees}`，召回后直接定位

### 6.3 多路召回
| 路 | 用途 | 实现 |
|---|---|---|
| Dense（BGE-M3） | 语义相近 | Milvus IVF_PQ |
| Sparse（BM25） | 关键字命中（如 "fat32", "buddy"） | Elasticsearch |
| **Call Graph MinHash** ⭐ | 抓"结构同源"（核心创新） | datasketch |
| Commit Message 聚类 | 抓"开发轨迹相似" | HDBSCAN |

> **Call Graph MinHash** 是查重的杀手锏：把每个函数的 (入度,出度,调用者签名) 作为 shingle，对整个仓库出一个 sketch。即使变量名全改，调用结构仍能命中。

## 7. 抗"幻觉"的工程化清单

| 风险点 | 对策 |
|---|---|
| LLM 编造"未实现的特性" | 事实表白名单约束输出 |
| LLM 把模板代码当作创新 | 内置 rCore / xv6 等基线模板指纹库，先剥离再分析 |
| 查重误报（同模板但独立写）| MinHash + LLM 双判，不一致打 "需人工" |
| 长上下文遗忘 | 关键证据用 retrieval 重新注入，不全塞进 prompt |
| 引用伪造 | Writer 输出引用后，由独立 verifier 重新打开文件比对，不一致即拒 |

## 10. 评测方案（怎么证明"靠谱"）

| 维度 | 指标 | 方法 |
|---|---|---|
| 描述准确性 | 人工抽查 20 个仓库的 Project Card，逐条核对引用 | 准确率 ≥ 95% |
| 查重召回 | 人工构造 10 对"已知抄袭"样本，看是否能召回 | Recall@5 ≥ 0.9 |
| 查重精度 | 人工标注 30 对"非抄袭但同模板"，看误报率 | FPR ≤ 0.1 |
| 抗幻觉 | 在 Project Card 中随机抽 100 条结论，全部需有可验证引用 | 引用真实率 100% |
| 时延 | 单仓库端到端分析时间 | ≤ 5 min（缓存命中 ≤ 30s） |
